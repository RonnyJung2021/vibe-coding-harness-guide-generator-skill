# Harness 工程

Harness 工程是构建**环境系统**的学科，这套系统使 AI Agent 能够在生产环境中被可靠控制并稳定运行。

**核心公式**: `Agent = Loop(Model + Harness)`

## 八大要素

每个 Harness 系统必须解决以下八个问题：

| # | 要素 | 核心问题 |
|---|---------|--------------|
| 1 | **任务表达** | 任务如何被声明？输入/输出契约必须无歧义。 |
| 2 | **上下文组织** | Agent 接收什么上下文？只传递与任务相关的信息。 |
| 3 | **工具治理** | 工具如何被发现、验证和拦截？ |
| 4 | **状态管理** | 状态如何被保存、恢复和裁剪？ |
| 5 | **反馈回路** | 执行反馈如何回流给模型？ |
| 6 | **错误分类** | 错误如何被分类、重试或升级？ |
| 7 | **安全边界** | 危险操作如何在运行时被阻止？ |
| 8 | **验证机制** | 系统如何验证任务实际完成了？ |

## 五层架构

### 第一层：上下文装配系统

Prompt 不是静态文档——它是**动态装配系统**。

按优先级和新鲜度组织上下文：

1. 基础模板
2. 用户偏好（SOUL.md）
3. 项目上下文（AGENTS.md）
4. Skills
5. 工具描述
6. 工具定义 JSON

**核心原则**：不同信息来源有不同的优先级、新鲜度和职责。不要把所有内容都塞进一个大 prompt。

### 第二层：工具治理系统

工具一旦引入，复杂度会立即上升。每个工具系统背后都有四个问题：

1. **发现** — 模型如何找到工具？
2. **验证** — 参数如何被检查？
3. **风险分级** — 工具如何按风险分级（读/写/执行/危险）？
4. **拦截** — 调用如何被统一拦截和审计？

要构建一个工具系统，而不是工具列表。真正的工具系统必须是可控、可审计、可演化的。

### 第三层：安全与审批系统

一旦 Agent 能写文件、执行 shell 命令或修改配置，它就不再是"聊天模型"——它是**操作系统参与者**。

设计三道防御层：

1. **子进程管理**：会话池、数量限制、输出截断、超时终止、自动清理
2. **命令守卫**：命令解析、危险命令黑名单、system hint 回传
3. **审批系统**：风险分级、审批模式、指纹缓存、危险模式兜底

**核心原则**：安全不能依赖"模型会守规矩"。真正的安全边界必须落在运行时。

### 第四层：反馈与状态系统

Agent 之所以能持续做事，不是因为"会思考"，而是因为**不断收到反馈**：

- 工具执行结果
- 审批是否通过
- 输出是否被截断
- 查找结果是否过多
- 命令是否被拒绝
- 当前任务计划是否已更新
- 会话是否可恢复

把系统内部发生的事翻译成**模型能消费的反馈语言**。没有这层翻译，模型只能看到"失败了"；有了这层翻译，模型才能理解原因并采取纠正行动。

### 第五层：熵管理系统

Agent 一旦持续运行，系统不可避免地会积累"脏污"：

- Prompt 越来越长
- 规则越来越旧
- 上下文越来越臃肿
- 工具定义越来越多但没人维护
- 会话历史越来越重
- 记忆混入大量低价值信息

长期 Harness 需要持续的**熵管理**——让过期规则失效、压缩长上下文、清除低价值历史、以可持续方式沉淀项目知识。

## 执行清单

将 Harness 工程应用于新项目或功能时，验证以下内容：

### 任务定义
- [ ] 输入契约明确（Agent 收到什么？）
- [ ] 输出契约可验证（如何判断成功了？）

### 上下文
- [ ] 上下文按优先级装配，而非堆砌
- [ ] 只向下游传递与任务相关的上下文
- [ ] 排除噪声（导航、广告、脚本）

### 工具
- [ ] 工具以 DSL 风格声明
- [ ] 执行前验证参数
- [ ] 风险分级并强制执行
- [ ] 调用可被拦截和审计

### 状态
- [ ] 中间产物被持久化（不仅是最终输出）
- [ ] 状态可从检查点恢复
- [ ] 状态臃肿时进行裁剪

### 反馈
- [ ] 系统错误被翻译为模型可读反馈
- [ ] 模型理解失败原因，而非仅知道失败

### 错误处理
- [ ] 错误被分类（网络、依赖、权限等）
- [ ] 临时失败有重试逻辑
- [ ] 失败升级到适当的环境/能力层级

### 安全
- [ ] 危险操作在运行时被阻止，而非仅在 prompt 中警告
- [ ] 子进程执行有限制（数量、超时、清理）
- [ ] 写操作限定在工作区范围内

### 验证
- [ ] 任务完成可验证（文件存在、结构匹配、内容存在）
- [ ] 证据被持久化（日志、产物、快照）

## 核心哲学

> **Harness = 使 Agent 可被控制并稳定操作的环境系统**

最重要的词不是"Agent"，也不是"Model"。而是：

- **约束** — Agent 必须在边界内运行
- **反馈** — Agent 必须理解结果，而不仅是输出
- **稳定** — 系统必须可维护、可恢复

Agent 的能力不只是模型能力——而是 `Model × Environment`。

同样的模型，在不同系统中，可能产生截然不同的结果。

**环境是可复用资产。模型可以替换，但 Harness 不可跳过。**

## 何时调用

在以下情况应用此 skill：

1. 从零开始构建新的 Agent 项目
2. 实现需要工具调用、上下文管理或状态持久化的新功能
3. 为 Agent 操作设计反馈循环和错误处理
4. 为可修改文件或执行命令的 Agent 建立安全边界
5. 构建需要管理中间状态的多步骤流水线

此 skill 应在**项目初始化时**调用，而非事后补救。
